## 1. Title

**Predictive Resource Allocation in Serverless Database-as-a-Service via Lightweight Reinforcement Learning**

This repository accompanies a PeerJ Computer Science AI Application
submission. It contains the synthetic multi-tenant workload generator,
the database-as-a-service environment simulator, the three reinforcement
learning agents (Linear-Q, Deep Q-Network, Federated Linear-Q), the
three non-RL baselines (Threshold, Over-Provision, ARIMA), the training
and evaluation pipeline, the ablation experiments, the inference
benchmark, the figure-generation scripts, the smoke tests, and the
manuscript LaTeX source.

## 2. Description

### What this project is

A complete, reproducible artefact for evaluating a **tile-coded linear
Q-learning** controller against a small **deep Q-network** and three
non-RL baselines (threshold autoscaler, fixed over-provisioning, ARIMA
trend predictor) on a synthetic but realistically-structured
multi-tenant serverless DBaaS workload.

### What problem it addresses

Serverless DBaaS platforms (Amazon Aurora Serverless v2, Google
AlloyDB, Azure SQL Hyperscale) scale capacity in seconds in response
to demand. The autoscalers shipped with these platforms are reactive
and threshold-based, which causes cold-start latency under bursty
multi-tenant traffic and silent over-provisioning during quiet periods.
Predictive machine-learning forecasters can pre-empt these failures
but are typically too computationally heavyweight to run continuously
inside the control plane.

### What the controller does

At each minute, the controller observes a six-dimensional state
(windowed mean QPS, windowed peak QPS, windowed linear-trend slope of
QPS, current capacity in ACU, normalised minute-of-day, current
utilisation) and chooses one of eight discrete Aurora-style capacity
levels: 0.5, 1, 2, 4, 8, 16, 32, or 64 ACU. The reward jointly
penalises SLA-violation severity, monetary cost, and cold-start
overhead.

### Key headline numbers (held-out 3-day trace, seed 137)

| Agent | SLA viol. | Mean lat. (ms) | P99 lat. (ms) | Mean ACU | Cost  |
| ---: | ---: | ---: | ---: | ---: | ---: |
| LinearQ (ours) | **3.6 %** | 83.1 | 1\,427 | 32.2 | 13\,919 |
| FederatedLinearQ | 10.7 % | 91.7 | 1\,427 | 27.2 | **11\,752** |
| DQN | 8.8 % | 54.6 | 113 | 32.0 | 13\,821 |
| Threshold | **0.2 %** | 48.3 | **77** | 35.5 | 15\,354 |
| ARIMA | 2.6 % | 60.6 | 108 | **27.4** | 11\,834 |
| OverProvision | 0.0 % | 29.8 | 44 | 64.0 | 27\,642 |

### Honest caveat

When LinearQ does violate the SLA the tail latency is severe
(saturated at $\approx 1\,427$ ms). Operators with hard tail
constraints should mitigate this via reward-weight tuning, a
constrained action set, or a safety-net wrapper (see Manuscript
section *Failure Mode Analysis*).

## 3. Dataset Information

### Provenance

All datasets in this repository are **synthetic**. They are produced
by a parametric multi-tenant workload generator (`src/workload_generator.py`)
that combines diurnal seasonality, weekly modulation, slow drift,
Poisson micro-bursts, and Pareto-tailed flash crowds. Parameters were
informed by public engineering documentation about typical SaaS
workload shapes; **no real customer trace was used or replayed**.

The full mathematical specification of the generator is in the
manuscript, equation (1). Tenant profiles (baseline QPS, diurnal
amplitude, burst statistics, SLA target) are listed in the
manuscript's Table 2.

* `minute` — integer index, one row per minute of simulated time, starting at 0.
* `tenant_id` — string identifier (`t01_ecommerce`, …, `t08_ml_pipeline`).
* `workload_type` — categorical: `OLTP`, `OLAP`, `Mixed`.
* `sla_ms` — per-tenant SLA target in milliseconds (informational; the RL agents optimise a cluster-wide SLA at 100 ms).
* `qps`, `total_qps` — queries per second at that minute (float).
* `cpu_pct`, `mem_pct` — synthetic utilisation in percent (float; informational only — RL state uses utilisation derived from QPS/ACU).
* `latency_p99_ms`, `mean_latency`, `max_latency` — synthetic per-tenant 99th-percentile latency in milliseconds (informational — actual latency seen by the agent is recomputed by the M/M/c-style simulator).
* `mean_cpu`, `max_cpu`, `n_tenants` — minute-aggregate cluster statistics.

* Federated client seeds: `42 + 1000·(i+1)` for `i ∈ {0,1,2,3}`.

**Run an environment step**
```python
import pandas as pd
from src.environment import DBaaSEnv, EnvConfig

workload = pd.read_csv("data/workload_aggregate.csv")
env = DBaaSEnv(workload_df=workload, config=EnvConfig())
obs = env.reset()
next_obs, reward, done, info = env.step(action_idx=4)  # action 4 = 8 ACU
```

**Train a Linear-Q agent**
```python
from src.agents import LinearQAgent, LinearQConfig

agent = LinearQAgent(
    obs_low=[0, 0, -50, 0.5, 0.0, 0.0],
    obs_high=[2000, 2000, 50, 64.0, 1.0, 1.0],
    n_actions=8,
    config=LinearQConfig(alpha=0.25, gamma=0.9, bins=4, num_tilings=3),
)

obs = env.reset()
for t in range(720):
    action = agent.act(obs)                   # eps-greedy
    next_obs, reward, done, _ = env.step(action)
    agent.update(obs, action, reward, next_obs, done)
    obs = next_obs
```

### Random seeding strategy

Every module that uses randomness accepts an explicit `seed` parameter
that is passed through to a private `numpy.random.Generator`. There
are no module-level globals; consequently running the same script
twice with the same seed produces byte-identical output. The four
federated clients receive distinct seeds `42 + 1000·(i+1)` for
`i ∈ {0,1,2,3}` so that they explore differently.

## 7. Methodology

This section summarises the data-processing and modelling steps. The
manuscript's *Materials \& Methods* section contains the full formal
specification with equations.

### 7.1 Data preprocessing

The workload generator produces three CSV files per run:

1. **Per-tenant traces** (`workload_<tenant>.csv`) — one row per
   simulated minute per tenant, with columns `minute`, `tenant_id`,
   `workload_type`, `sla_ms`, `qps`, `cpu_pct`, `mem_pct`,
   `latency_p99_ms`.

2. **Combined trace** (`workload_combined.csv`) — concatenation of
   all per-tenant rows. Equivalent to a `UNION ALL` over the
   per-tenant CSVs.

3. **Aggregated trace** (`workload_aggregate.csv`) — produced by a
   `groupby('minute').agg(...)` reduction of the combined trace with
   columns `minute`, `total_qps`, `mean_cpu`, `max_cpu`,
   `mean_latency`, `max_latency`, `n_tenants`. **This aggregated
   trace is the input consumed by the RL training pipeline.**

The training set is a single 7-day run with seed 42
($T_{\text{train}} = 10\,080$ minutes). A separate held-out
evaluation set is generated with seed 137 over a 3-day horizon
($T_{\text{eval}} = 4\,320$ minutes) to test out-of-distribution
generalisation. The two seeds produce QPS distributions that
overlap substantially but are not identical (min/max/mean QPS:
training = 65 / 1\,720 / 691; evaluation = 131 / 1\,467 / 754).

The agent's six-dimensional state vector is computed **online** from
`workload_aggregate.csv` via a 10-minute trailing window that
produces windowed mean QPS, windowed maximum QPS, and windowed
linear-trend slope. **No additional smoothing, normalisation, or
feature engineering is applied beyond the tile coder's per-dimension
min-max scaling** described below. All values are stored as 32-bit
floats. No data augmentation, no train-time shuffling, and no
preprocessing of the evaluation trace are applied, to ensure that
evaluation reflects the agent's behaviour on realistically-streamed
minute-resolution data.

### 7.2 Tile-coder feature construction

The continuous 6-dimensional state is projected to a sparse
high-dimensional binary feature vector using $K = 3$ overlapping
uniform grids, each with $B = 4$ bins per dimension. For each
tiling $k$:

1. **Min-max scale** each state component to $[0, 1)$:
   $\tilde{s}_d = (s_d - s_d^{\min}) / (s_d^{\max} - s_d^{\min})$,
   clipped to $[0, 1 - 10^{-4}]$. The min and max values per
   dimension are constants chosen to cover the observed ranges.
2. **Shift** by a tiling-specific random offset drawn from
   $\mathrm{Uniform}(0, 1/B)^d$ and fixed at agent initialisation.
3. **Discretise** each shifted component into $B$ bins.
4. **Flatten** into a single tile index via radix-$B$ encoding.

The active feature vector for state $s$ has exactly $K = 3$ active
indices out of $M = K \cdot B^d = 3 \cdot 4^6 = 12\,288$. The
weight matrix is $\theta \in \mathbb{R}^{8 \times 12\,288}$,
totalling 98\,304 32-bit floats = 384 KB.

### 7.3 Training protocol

Each RL agent trains for 100 episodes of 720 minutes each
(= 72\,000 environment interactions). Exploration uses an
$\epsilon$-greedy schedule linear from $\epsilon_0 = 1.0$ to
$\epsilon_\infty = 0.05$ over the first 50% of training steps.

* **LinearQ**: one-step Q-learning update via the formula in the
  manuscript's equation (8), applied to the $K = 3$ active tile
  indices per action. No replay buffer.
* **DQN**: 64-unit ReLU hidden layer (968 parameters total), mean-
  squared TD loss with a target network synchronised every
  200 steps, replay buffer of 5\,000, mini-batch 32, Adam optimiser
  with learning rate $10^{-3}$.
* **FederatedLinearQ**: $N_c = 4$ clients running Algorithm 2 in
  the manuscript; coordinator FedAvg every $E_{\mathrm{agg}} = 5$
  episodes.

### 7.4 Evaluation protocol

After training, every agent is evaluated greedily on the full
3-day held-out trace (4\,320 minutes). The two RL agents continue
to update their value function on-line during evaluation, mirroring
continual learning in production. The non-RL baselines' `update`
method is a no-op.

Five metrics are reported per agent:

* **SLA violation rate** — fraction of minutes where realised
  latency exceeds the 100 ms SLA.
* **Mean latency** — arithmetic mean of per-minute P99 latency.
* **P99 latency** — 99th percentile of per-minute P99 latency.
* **Mean ACU** — average capacity allocated.
* **Cost** — cumulative scaled cost = $\sum_t c_t$ (recover
  absolute USD by dividing by 50).

Formal definitions appear in the manuscript's Definitions 1–5.

### 7.5 Ablations

`src/ablations.py` runs:

* **Tile granularity**: $B = 4$ vs $B = 8$ at fixed $K = 3$.
  Doubling $B$ multiplies parameters by $2^6 = 64$ and (in our
  experiments) hurts policy quality on the held-out trace.
* **Argmax tie-breaking**: random vs deterministic. With Gaussian
  init these are identical; the random tie-break matters only
  under constant initialisation.

Both ablation results land in `results/ablations.csv`.

### 7.6 Computational profiling

`src/benchmark_inference.py` measures wall-clock $\mu$s per
inference call for each agent, averaged over 50\,000 calls on a
single CPU thread.

## 11. Troubleshooting

### `ModuleNotFoundError: No module named 'numpy'` (or pandas, matplotlib)

Run `pip install -r requirements.txt`. If you are inside a virtual
environment, make sure it is activated. If `pip` is associated with
Python 2, use `pip3` or `python -m pip install -r requirements.txt`.

### Pipeline output differs slightly from the checked-in CSVs

* Confirm Python ≥ 3.10 (`python --version`).
* Confirm NumPy ≥ 1.26 (`python -c "import numpy; print(numpy.__version__)"`).
* The single biggest source of small numerical drift between platforms
  is the BLAS used by NumPy. Switching from OpenBLAS to MKL (or vice
  versa) can move the DQN's inference time by a factor of 2 and shift
  fractional metrics by approximately 0.1 percentage points. The
  *relative ordering* of agents should be invariant.

### LaTeX compilation fails with "wlpeerj.cls not found"

The included `paper/wlpeerj.cls` is a shim. It is in the same
directory as `main.tex`, so `pdflatex main.tex` from inside `paper/`
should find it. For final PeerJ submission, replace this shim with
the official PeerJ class downloadable from
<https://peerj.com/about/author-instructions/>.

### LaTeX compilation fails with "missing font" warnings

Install `lmodern` and `texlive-fonts-extra`:

```bash
sudo apt-get install lmodern texlive-fonts-extra texlive-latex-extra texlive-science
```

### "Killed" / out of memory

Should not happen — peak memory is around 500 MB during DQN
training. If it does, you may be running on a memory-constrained
container; lower `EPISODES`:

```bash
make train EPISODES=20
```

### Smoke tests pass but `make train` errors

Run with verbose output to see where it fails:

```bash
python -u src/train.py --workload data/workload_aggregate.csv \
                       --eval-workload data/workload_aggregate_eval.csv \
                       --out results --episodes 5 --episode-length 240 --seed 42
```

If the failure is in `make data`, regenerate the CSVs explicitly:

```bash
rm -rf data
make data
```
